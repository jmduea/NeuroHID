<task type="auto">
  <name>Task 2: Recording tap task and integration</name>
  <files>crates/neurohid-core/src/tasks/recording.rs, crates/neurohid-core/src/tasks/mod.rs, crates/neurohid-core/src/service.rs, crates/neurohid-types/src/control.rs</files>
  <action>
Implement the recording tap and wire it into the service:

1. Control protocol: In neurohid-types/src/control.rs add ControlCommand variants StartRecording { output_path: Option<std::path::PathBuf> } and StopRecording. Add ControlResponsePayload variant RecordingStarted { session_id: String, output_path: String } and RecordingStopped { session_id: String } (and/or surface errors via existing Error variant). Snapshot should expose recording_active: bool and current_session_id: Option<String> so CLI/Hub can show feedback.

2. Recording task (crates/neurohid-core/src/tasks/recording.rs):
   - Receives: clone of sample_broadcast_tx (subscribe), clone of action_broadcast_tx (subscribe), config (RecordingConfig + system config for snapshot), channel for start/stop commands (output_path override, stop).
   - On StartRecording: create session folder (session_&lt;id&gt; with id = e.g. now_micros()), write manifest.json (started_at, format_version, runtime_version), write config snapshot (copy of SystemConfig or path to --config), write profile_meta.json if profile active. Open streams/ and actions.jsonl for writing. Do not block the pipeline: receive samples and actions on dedicated loop (tokio::select with shutdown and command channel); write samples to streams (e.g. one file per source_id or combined with stream index), append actions to actions.jsonl (one JSON line per action with timestamp, decision_id, confidence, mouse/keyboard summary). On StopRecording: set ended_at in manifest, flush and close files, respond. On failure: stop recording, keep what was written, surface error (log and optionally set state so snapshot/API can report it). Enforce optional cap: max_duration_secs / max_size_mb from config when set.
   - Run as separate task; do not block sample_tx or decoder path. Use buffered/async writes where reasonable.

3. Service integration (service.rs): When building the service, create a channel for recording commands. Spawn RecordingTask with sample_broadcast_tx.subscribe(), action_broadcast_tx.subscribe(), config.recording, and command rx. Handle ControlCommand::StartRecording / StopRecording in the control handler: send to recording task and return RecordingStarted/RecordingStopped or Error. When auto_mode is TiedToRuntime or TiedToOutput, start recording on runtime start or when output is enabled (per config) and stop on shutdown or output disabled — implement by sending start/stop from the same place that knows runtime/output state.

4. Export RecordingTask (and any public types) from tasks/mod.rs.
  </action>
  <verify>`cargo check -p neurohid-core` and `cargo check -p neurohid-types` pass. Control commands StartRecording/StopRecording exist. Snapshot includes recording state. Recording task is spawned in service and receives samples/actions from broadcast; session folder created with manifest, config, streams, actions.jsonl on start; stop writes ended_at and flushes.</verify>
  <done>Recording starts/stops via control; session folder contains manifest, config snapshot, streams, actions.jsonl; on failure recording stops and keeps written data; optional cap enforced.</done>
</task>
