  <action>
Add Device CLI and unified neurohid entrypoint per CONTEXT and RESEARCH.

In neurohid-service.rs:
- Add CliCommand variant Device { #[command(subcommand)] command: DeviceCommandCli }.
- Add DeviceCommandCli: List { optional --json, -q, -v; optional --endpoint for control server }, Connect { --device-id &lt;id&gt; OR --criteria &lt;type&gt; (e.g. LSL), optional --endpoint }. Default output: human-readable table of streams (id, name, type, channels); --json: compact one-line JSON to stdout. Progress/spinner on stderr; -q suppresses progress. Exit codes: 0 success, 1 generic error, 2 not found, 3 config invalid (document in code or docs). When multiple devices and no --device-id and interactive TTY: prompt to choose; otherwise no prompts so scripts work. Use existing send_control_request_blocking with RescanStreams then Snapshot for list; then ConnectStream for connect. --endpoint defaults to 127.0.0.1:47384.
- Global or command-level flags: --config, --profile per CONTEXT; -v/--verbose, -q/--quiet; --json only on commands that produce list/object (e.g. device list).

In neurohid.rs (Hub binary):
- Before starting the Hub, parse argv: if argc >= 2 and argv[1] is one of device, config, pipeline, control, daemon, then locate neurohid-service binary (e.g. same directory as current exe or env/cargo convention) and exec/spawn it with argv[1..], exit with its exit code. This makes "neurohid device list" and "neurohid device connect ..." work with a single entrypoint name. On Windows use std::process::Command; on Unix consider std::os::unix::process::CommandExt::exec if same process is preferred, or Command otherwise. Ensure no GUI or heavy init runs when dispatching to CLI.
