  <action>
Add a `Control` variant to `CliCommand` so that `neurohid-service control &lt;subcommand&gt;` sends control requests to a running service (client path only; no runtime started).

- Add enum `ControlCommandCli` with subcommands: `Snapshot`, `SetOutputEnabled { enabled: bool }` (use clap ValueEnum or bool for enabled). Add `Control { command: ControlCommandCli, endpoint: String }` to `CliCommand`. Endpoint: `--endpoint &lt;addr&gt;` (e.g. 127.0.0.1:47384), required for control subcommand. Default endpoint 127.0.0.1:47384 is acceptable as a convenience.
- In main(), when `args.command` is `Some(CliCommand::Control { command, endpoint })`, build `RuntimeIpcConfig { transport: RuntimeIpcTransport::TcpLoopback, endpoint, ..default() }`, then dispatch: for Snapshot call neurohid_ipc::send_control_request_blocking (or existing send_control_request once then block) with ControlCommand::Snapshot, parse response to get ControlSnapshot, and print human-readable status (device_connected, decoder_ready, output_enabled, pipeline_integrity_degraded / integrity_issue_count as in research). For SetOutputEnabled { enabled } send ControlCommand::SetOutputEnabled { enabled }, then print success or error. Use neurohid_types ControlRequest/ControlResponse/ControlResponsePayload. No new dependencies; neurohid-ipc already exports send_control_request_blocking.
- If the binary is built without a tokio runtime for the control path, use send_control_request_blocking; if control path runs inside tokio::main, send_control_request_once with .await is fine. Prefer blocking for a simple one-shot CLI so the process exits after the request.
- Follow crates/AGENTS.md: Result/?, no unwrap in library paths; keep control CLI code in the binary.
  </action>
